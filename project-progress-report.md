**วันที่รายงาน:** 28 พฤษภาคม 2569

# 1. ภาพรวมการดำเนินงาน
โครงการระบบประสานงานและส่งมอบเวรทางห้องปฏิบัติการ โรงพยาบาลสังขะ ได้ดำเนินการพัฒนาโครงสร้างหลักของระบบเสร็จเรียบร้อยแล้วในระดับใช้งานจริงเบื้องต้น โดยครอบคลุมทั้งส่วน User Portal, Admin Portal, ระบบสิทธิ์ผู้ใช้งาน และระบบเชื่อมต่อฐานข้อมูล

ปัจจุบันอยู่ในระยะเก็บรายละเอียดเพิ่มเติม ปรับปรุงประสิทธิภาพ และเตรียม Deploy ขึ้นระบบ **Cloudflare Pages** เพื่อเข้าสู่ระยะทดสอบใช้งานจริงภายในหน่วยงาน

นอกจากนี้ ในระหว่างการพัฒนามีการปรับเปลี่ยนสถาปัตยกรรมบางส่วนจากแผนเดิม เพื่อให้เหมาะสมกับโครงสร้างความปลอดภัย ความเร็ว และความยืดหยุ่นในการดูแลระบบในระยะยาว โดยเฉพาะการ**ย้ายระบบอัตโนมัติจาก Make.com มาใช้ Supabase Edge Functions (Direct serverless API)** ซึ่งส่งตรงไปยัง LINE Messaging API โดยไม่ต้องพึ่งพาแพลตฟอร์มรวมข้อมูลตัวกลางจากภายนอก ช่วยปิดช่องโหว่ความลับข้อมูลคนไข้รั่วไหลได้อย่างสมบูรณ์

# 2. การเปรียบเทียบสถาปัตยกรรมระบบ (Architecture & Tech Stack)

| ส่วนระบบ | แผนเดิม | ระบบที่ใช้งานจริง | หมายเหตุ / ประโยชน์ทางเทคนิค |
| :--- | :--- | :--- | :--- |
| **Frontend** | Next.js 16 App Router | **React 18 + Vite (TypeScript)** | ปรับเพื่อเพิ่มความเร็วในการดาวน์โหลดแอปหน้าเดียว (SPA) และลดความซับซ้อนของ Routing |
| **Backend** | Supabase + RLS + RPC | **Supabase + RLS + RPC + Realtime** | ดำเนินตามแผน พร้อมเปิดใช้งาน Realtime Sync เพื่อส่งเวรปั๊บอัปเดตบนหน้าจอทันที |
| **Automation** | Make.com + LINE Messaging API | **Supabase Edge Functions + LINE API** | **ยกระดับความลับสูงสุด:** พัฒนาระบบ Serverless Edge Functions บน Supabase ทำงานร่วมกับ Database Trigger ส่งข้อมูลเข้า LINE Group โดยตรง ปราศจากการส่งถ่ายผ่านแพลตฟอร์มภายนอก (Make.com ถูกตัดออก 100%) |
| **Hosting** | Vercel | **Cloudflare Pages** | ปรับปรุงเพื่อความปลอดภัยสูงสุด รองรับ Private Repository และการผูกโดเมนภายในเครือข่ายโรงพยาบาลได้สะดวก |

*หมายเหตุ: สำหรับตัวเซิร์ฟเวอร์ฐานข้อมูลหลังบ้าน Supabase โฮสต์อยู่ที่ภูมิภาค **Oceania (Sydney) Region: ap-southeast-2** เพื่อประสิทธิภาพและความเสถียรในการทำงานร่วมกับโครงสร้างพื้นฐานเดิมของทีม*

# 3. สถานะการดำเนินงานเทียบตาม Sprint

## Phase 1 MVP

### Sprint 1 — Database & Public Homepage
*   **สถานะ:** 🎉 เสร็จสมบูรณ์ 100%
*   **ดำเนินการแล้ว ได้แก่:**
    *   ออกแบบ Database Schema หลัก (`handover_sys` tables, views, enums)
    *   ตั้งค่าความปลอดภัยระดับแถวข้อมูล (Row Level Security - RLS) เพื่อป้องสิทธิ์บุคคลภายนอกอย่างเคร่งครัด
    *   ระบบตรวจสอบ PIN ของผู้ส่งเวรเป็นรายบุคคล (PIN Verification RPC)
    *   ระบบสุ่มออกเลขที่งานอัตโนมัติ (Task Number Generator e.g., LAB-0001)
    *   หน้าบ้านส่งฟอร์มเวรด่วนสำหรับส่งข้อมูลด่วนแบบไม่ต้องจำเข้าระบบ (PIN Verify & Send)
    *   ระบบดึงแสดงผลประวัติเวรส่งมอบย้อนหลัง 3 วันสำหรับบุคลากรสาธารณะ
*   **เพิ่มเติมจากแผนเดิม:** เพิ่มระบบพรางข้อมูลผู้ป่วยอัตโนมัติบน Public View เพื่อไม่ให้ข้อมูลส่วนบุคคลปรากฏสู่ภายนอกก่อนได้รับสิทธิ์กุญแจความยินยอม

---

### Sprint 2 — Authentication & User Dashboard
*   **สถานะ:** 🎉 เสร็จสมบูรณ์ 100%
*   **ดำเนินการแล้ว ได้แก่:**
    *   ระบบเข้าสู่ระบบของเจ้าหน้าที่ (Login) โดยใช้ Dropdown บัญชีชื่อพนักงาน + PIN รหัสส่วนตัว (ให้ประสิทธิภาพ UX ดีกว่าการพิมพ์ Email ตามแผนเดิม)
    *   ระบบ Middleware แยกสิทธิ์ขาดและจำกัดการเข้าถึง (Route Guards: Admin / User / Public)
    *   แดชบอร์ดสรุปกิจกรรมการส่งมอบเวรของเจ้าหน้าที่ห้องแล็บแบบ Realtime
    *   ระบบสิทธิ์ในการกดยอมรับงาน (Accept Handover Case) เพื่อยืนยันการรับไม้ต่องานเวรสมบูรณ์

---

### Sprint 3 — LINE Integration & Automation (Serverless Direct)
*   **สถานะ:** 🎉 เสร็จสมบูรณ์ 100%
*   **ดำเนินการแล้ว ได้แก่:**
    *   เชื่อมต่อตรงผ่าน LINE Messaging API ด้วย Supabase Edge Functions ไม่พึ่งเว็บบล็อกนอก
    *   ระบบสร้าง Flex Message การแจ้งเตือนที่สวยงาม ชัดเจน และแจกแจงวิกฤติ (Critical, Normal, Blood Bank) ส่งอัตโนมัติเข้ากลุ่มไลน์เวรแล็บทันทีที่มีข้อมูลลงทะเบียนใหม่
    *   ระบบบันทึกรายการลงนามผู้รับไม้ต่อเวร (Sign-off Logging Process)
    *   ระบบป้องกันการกดรับงานทับซ้อน (เมื่อผู้ปฏิบัติรับงานอื่นรับงานเดิมไปแล้ว ระบบจะปฏิเสธการเซ็นต์ชื่อซ้ำทางไลน์)
    *   เชื่อมต่อ Endpoint Webhook แบบเสถียรรักษาความปลอดภัย Header Secret

### Sprint 4 — Admin Panel & UI Refinement
*   **สถานะ:** ⏳ อยู่ระหว่างดำเนินการ (ประมาณ 80%)
*   **ดำเนินการแล้วบางส่วน ได้แก่:**
    *   ระบบควบคุมและหน้าจอของพอร์ทัลแอดมิน (Admin Dashboard Showcase)

    *   หน้าระบบควบคุมกลุ่มไลน์แล็บ (LINE Settings Group Management Area)

    *   ขัดเกลากลไกบันทึกและจัดการกลุ่มพนักงาน Senders' Management (PIN reset, Toggle activation)
    *   ปรับปรุง Transition การเปลี่ยนสถานะข้อมูลเวรของตู้ประสานงานให้แสดงลื่นไหลลุคพรีเมียม

### Sprint 5 — Production Deployment
*   **สถานะ:** ⏳ เตรียมดำเนินการ (Ready to Deploy)
*   **ปรับแผนการใช้งาน:** เปลี่ยนแปลงจุดปลายโฮสติ้งจาก Vercel ไปใช้งานบนระบบ **Cloudflare Pages** เพื่อรองรับความมั่นคงปลอดภัยและความเสถียร
*   **ดำเนินการเตรียมพร้อมแล้ว:**
    *   จัดทำดักจับกระจายโครงสร้าง Routing ด้วยไฟล์ `/public/_redirects` ป้องกันปัญหา Error 404 ยามผู้ใช้งานกดรีเฟรชเบราว์เซอร์
    *   เตรียมชุด Environment Variables ปลายทาง (`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`) เพื่อไปเริ่มใช้งานบน Cloudflare Pages
    *   จัดสัดส่วนเตรียมอัปเดตเข้าสู่ Private/Security Git Repository บน GitHub เพื่อรักษามาตรการความลับนักพัฒนา

# 4. งานที่ดึงมาดำเนินการก่อนแผนเดิม (Proactive Milestones Covered)

เนื่องจากระบบนี้ประมวลผลข้อมูลคนไข้อันเป็นสิทธิประโยชน์ด้านข้อมูลอ่อนไหว (Sensitive Data) ทางโครงการจึงตอบรับมาตรฐานความปลอดภัยอย่างเด็ดขาด โดยดึงงานสำคัญจาก Phase ระยะยาวมาสอดแทรกเพื่อความถูกต้องตามหลักนโยบายทางการแพทย์และกฎหมาย:

## 4.1 การจัดทำและปฏิบัติตาม พ.ร.บ. คุ้มครองข้อมูลส่วนบุคคล (PDPA Compliance)
*   ยึดเกณฑ์อ้างอิงของกระทรวงสาธารณสุขและหนังสือแจ้งประสาทบริการทางการแพทย์ (สป.สธ.) ล่าสุด
*   **ระบุฐานกฎหมายจำเพาะเจาะจง:** ใช้**ฐานภารกิจของรัฐ (Public Task)** ตามระเบียบมาตรา 26(5)(ข) ร่วมกับฐานการรักษาพยาบาล (Medical Treatment Basis) เพื่อการติดตามผู้ป่วยวิกฤติต่อเนื่อง โดยไม่จำเป็นต้องขอเอกสารยื่นยินยอมแยกย่อยจากผู้ใช้บริการทุกครั้ง แต่เข้มงวดสิทธิ์พนักงานผู้มีกลไกกุญแจสิทธิ์รักษาพยาบาลเท่านั้น
*   **ระบุอายุข้อมูล:** ข้อมูลคนไข้และประวัติเวรจะถูกบันทึกและจัดเก็บไว้บนระบบสำรองปลอดภัยเป็นระยะเวลารักษาสิทธิประโยชน์คนไข้**สูงสุดไม่เกิน 10 ปี** ตามกรอบมาตรฐานสถานพยาบาลของ สป.สธ.
*   **ระบบพรางข้อมูลผู้ป่วย (Automatic Data Masking):** เพิ่มระบบเปลี่ยนค่านิพจน์ (Regex Masking Engine) ในช่วงหน้าแสดงผลสาธารณะ โดยจะพรางรหัสประจักษ์ เช่น ตัวเลข HN และชื่อ-นามสกุลจริงผู้ป่วยเป็นตัวอักษรดอกจัน `****` ทันที ป้องกันบุคคลภายนอกมองผ่านจอ แต่จะเปิดเผยข้อมูลสมบูรณ์แก่เจ้าหน้าที่ผู้ปฏิบัติงานที่ล็อกอินผ่านระบบสิทธิ์ภายในและระบุ PIN ถูกต้องเท่านั้น

## 4.2 การศึกษาและเตรียมความพร้อม LINE LIFF
*   เริ่มวิเคราะห์โครงสร้างระบบเชื่อมโยงของ LINE LIFF (LINE Frontend Framework) ในระดับโครงสร้างเบสิกเพื่อให้พร้อมขยายแอปพลิเคชันลงบนมือถืออย่างเต็มสเกลในอนาคต

# 5. งานที่อยู่ระหว่างดำเนินการก่อนเปิดใช้งานจริง (Pending Launch Tasks)

เพื่อให้การเปิดใช้งานระบบประสานงานระดับสถาบันพยาบาลไม่มีความสะดุด ทีมวิศวกรและผู้เกี่ยวข้องจะดำเนินงานต่อดังนี้:

## 5.1 งานด้านระบบ LINE
*   ผูกเชื่อมระบบต่อประสาน LINE LIFF ปลายทางและขัดเกลาระบบ Postback ปุ่ม Quick Sign-off ทันทีที่มีการกดแจ้งเตือนจากหน้าแชต
*   ตรวจสอบเสถียรภาพของการตอบรับ Webhook จาก LINE Messaging API สูงสุด

## 5.2 งานนโยบายคุ้มครองความเป็นส่วนตัว (Privacy Policy)
*   จัดเตรียมร่างประกาศนโยบายการคุ้มครองข้อมูลส่วนบุคคลประจำระบบ ให้สอดรับโครงสร้างของโรงพยาบาลและยื่นเสนอทีมบริหารเซ็นต์อนุมัติใช้งานอย่างเป็นทางการ

## 5.3 งานปรับปรุงประสิทธิภาพหน้าจอ (UI Optimization)
*   ตรวจสอบ Responsive ความสะดวกในการสัมผัสใช้งานของพนักงานผ่านมือถือ iOS/Android (ความกว้างหน้าจอโทรศัพท์ช่วง 320px-480px)
*   ปรับ Navbar ด้านข้าง สไลด์ให้ลื่นไหล และปรับประสิทธิภาพการเขียนและดึง Query ฐานข้อมูล (DB View Indexes) เพื่อขยายความเร็วลด Latency

## 5.4 งาน Deploy ระบบจริง
*   ตั้งค่า Private Git Repository และโอนย้ายความเชื่อมโยงยิงขึ้นคลาวด์จริงสิทธิ์ Cloudflare Pages
*   ดำเนินการทดสอบขั้นตอนสุดท้ายและการสร้างความแข็งแกร่ง (Final Production End-to-End Testing)

# 7. บทสรุปความพร้อมใช้งานของระบบปัจจุบัน

สรุปในภาพรวม ระบบส่งเวรแล็บบอร์ดเจ้าหน้าที่โรงพยาบาลสังขะ ได้พัฒนาเสร็จสมบูรณ์รุดหน้าแล้วในส่วนกลไกวิศวกรรมหลัก (Core Engine), ฐานข้อมูลทนทานกาลเวลาพร้อม Realtime, ตลอดจนระบบเชื่อมไลน์ Edge Functions ไร้คนกลาง ปลอดภัย 100% 

ขั้นตอนที่รอคอยการส่งมอบเป็นเพียงสเตปการปรับขอบความประณีตของสปีดความเร็ว, การทดสอบการเข้าหน้าจออุปกรณ์มือถือให้สมสัดส่วน, การเรียบเรียงนโยบายความลับ และเปิดโฮสติ้งจริง คาดการณ์ว่าระบบประสานงานและส่งเวรของห้องปฏิบัติการ โรงพยาบาลสังขะ จะพร้อมยืนเวรอย่างเป็นทรงชัยสงครามสำเร็จลงตัวได้ภายในเวลา 2–3 สัปดาห์ข้างหน้านี้อย่างสมบูรณ์แบบสูงสุด

---
**หมายเหตุ:** เอกสารเปรียบเทียบแผนงานฉบับปรับปรุงนี้ จัดทำขึ้นเพื่อเป็นข้อมูลระดับปฏิบัติการภายใน กลุ่มงานเทคนิคการแพทย์ โรงพยาบาลสังขะ เพื่อใช้รายงานความก้าวหน้าและการบริหารจัดการทรัพยากรผู้พัฒนาซอฟต์แวร์เป็นสำคัญ
